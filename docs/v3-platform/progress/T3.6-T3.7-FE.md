### [12:14:00] Step 5: T3.8-FE decision — no-op (justified)
**Action:** Plan §2904 prescribed `frontend/src/i18n/{en,it}.json` keys mirroring `lang/{en,it}/kb.php`'s refusal sub-tree. **Decided to skip this step.**

Rationale:
- The frontend has NO existing i18n library (no `useTranslation` consumer).
- The BE already returns the localized `answer` string (per T3.8-BE's `localizedRefusalMessage()`).
- Adding JSON files without a consumer would create dead weight.
- Adding an i18n library + wiring would create TWO drift-prone translation surfaces (BE's `lang/*/kb.php` + FE's i18n JSON) for the same strings.
- The existing FE chat UI is EN-only for non-content strings (button labels, hints) — consistent with that.

**Decision:** the FE renders BE-provided localized strings verbatim. The component's `body` prop is the contract, and BE owns its localization. Documented in L22 (already shipped in M3 backend wave).

The "ConfidenceBadge tier labels" (`high`/`moderate`/`low`/`refused`) and `RefusalNotice` hints stay hardcoded EN. If a future FE-translation requirement lands, it'll be a separate task with its own scope (i18n library selection + wiring).
