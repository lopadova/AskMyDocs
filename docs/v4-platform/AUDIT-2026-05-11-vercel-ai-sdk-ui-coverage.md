### 1.1 Public hooks (from `@ai-sdk/react`)

| Hook | Purpose | Typical use |
|---|---|---|
| `useChat` | Manages chat state — messages, status, send/stop/regenerate, error — over a streaming transport. | The core hook for any chat UI; drives token-by-token rendering, optimistic placeholders, error surfaces. |
| `useCompletion` | Streams a single-prompt text completion (no multi-turn message store). | Inline rewrite, summarisation, completion-style autosuggest. |
| `useObject` | Streams a typed JSON object that progressively materialises as fields arrive (Zod-schema validated). | Forms generated by LLM, structured extraction, classification UIs, partial JSON renders. |

### 1.2 `useChat` options / helpers / status enum

| Surface | Member | Notes |
|---|---|---|
| Option | `id` | Stable chat key so the SDK internal store partitions state per conversation. |
| Option | `messages` | Initial message snapshot (the v6 rename of v2's `initialMessages`). |
| Option | `transport` | Custom transport (`DefaultChatTransport`, `TextStreamChatTransport`, `DirectChatTransport`). |
| Option | `experimental_throttle` | Coalesces UI updates during fast streams (ms). |
| Option | `onFinish` | Fires on terminal stream event; receives `{ message, messages, isAbort, isDisconnect, isError }`. |
| Option | `onError` | Fires on transport / parse / network error. |
| Option | `onData` | Fires for every data part (especially transient data parts that don't enter message history). Throwing aborts. |
| Option | `onToolCall` | Auto-handles client-side tool calls. |
| Option | `sendAutomaticallyWhen` | Predicate (e.g. `lastAssistantMessageIsCompleteWithToolCalls`) that triggers an automatic follow-up turn after tool results land. |
| Helper | `messages` | `UIMessage[]` with `parts` array. |
| Helper | `sendMessage(message, options?)` | Submit; second arg overrides headers/body/metadata per request. |
| Helper | `setMessages` | Direct cache mutation (delete / edit / restore). |
| Helper | `status` | `'submitted' | 'streaming' | 'ready' | 'error'`. |
| Helper | `stop()` | AbortController-backed cancel. |
| Helper | `regenerate()` | Re-runs the last assistant turn. |
| Helper | `error` | Last error object. |
| Helper | `addToolResult` / `addToolOutput` | Inject a tool execution result back into the stream. |
| Helper | `addToolApprovalResponse` | Approve / deny a pending tool execution. |
| Helper | `resumeStream` | Reattach to an in-flight server stream after reload / network drop. |

### 1.3 `UIMessage.parts` type literals (v6)

| Part type | Purpose | Typical use |
|---|---|---|
| `text` | Plain assistant / user text delta. | Body of the bubble. |
| `reasoning` | Model's chain-of-thought trace (Anthropic, DeepSeek, o1, etc.). | Collapsible "Thinking…" trace. |
| `tool-<toolName>` | Typed tool call with `input-streaming` → `input-available` → `output-available` / `output-error` state machine. | Generative UI — render `<Weather {...part.output} />` etc. |
| `dynamic-tool` | Tool whose name isn't statically known at compile time. | MCP-bridged tools, runtime registries. |
| `step-start` | Boundary marker between multi-step tool turns. | Visual divider between automated follow-up steps. |
| `source-url` | RAG citation pointing at a URL. | Inline citation chips. |
| `source-document` | RAG citation pointing at a document object. | Doc-level grounding (e.g. PDF page). |
| `file` | Inline file attachment (image, PDF, etc.). | Multimodal input + assistant-generated attachments. |
| `data-<name>` | Custom typed data part (persisted in `message.parts`). | App-specific structured signals — confidence, refusal, artifacts, etc. |
| Transient data (via `onData`) | Ephemeral, not added to `message.parts`. | Toast notifications, progress, transient debug. |

## Section 3 — Recommended v4.5+ adoption batches

### Tier 1 — High-leverage adoption (5 items)

These visibly transform the AskMyDocs chat into ChatGPT-grade UX with manageable scope.

1. **Wire `regenerate` + `branch` buttons** — both already exist in `MessageActions.tsx`; just thread `chat.regenerate` through `MessageBubble.tsx:162` and add a fork-conversation BE endpoint for branch. Scope: **1 PR, ~6h** (regenerate alone is ~1h; branch needs a BE conversation-fork route).
2. **`experimental_throttle: 50` on `useChat`** — one-line in `use-chat-stream.ts:262-280`. Coalesces token-stream reconciliation; visible smoothness gain on long answers + measurable INP improvement. Scope: **1 PR, ~2h** (including a Playwright perf assertion).
3. **Suggested follow-up actions (post-turn)** — replace the static three-prompt `EmptyThread` with a dynamic `SuggestedActions` component that renders 2-3 follow-up prompts after each assistant turn (BE returns them as a `data-suggestions` part). Mirrors `vercel/ai-chatbot/suggested-actions.tsx`. Scope: **1 PR, ~1 day** (FE + BE addition of `StreamChunk::dataSuggestions`).
4. **`resumeStream` + persistent stream IDs** — survive tab reload mid-stream by reattaching to the server-side stream. Requires BE work too (persistent stream ID + `/conversations/{id}/messages/stream/resume` endpoint). Scope: **multi-PR cycle, ~1 week** (BE first, then FE wiring).
5. **Inline message edit** — `setMessages` is the SDK enabler. User edits a previous prompt → previous assistant turn dropped → resubmit. Pattern is straightforward with `setMessages` (slice array before the edited index, then `sendMessage`). Scope: **1 PR, ~1.5 days**.

### Tier 2 — Nice-to-have polish (8 items)

1. **`source-document` rendering** — adapter extension in `message-shape-adapters.ts:202-219` to accept both `source-url` AND `source-document`. ~3h.
2. **Multimodal input (file/image attachment)** — Composer-side `<input type="file">` + `parts[].type === 'file'` rendering. ~2 days (BE upload, FE preview, multipart contract).
3. **Toast surface** — replace the inline `chat-composer-error` banner with a non-blocking toast for transient errors. Pull `vercel/ai-chatbot/toast.tsx` pattern. ~4h.
4. **Shimmer placeholder during `status === 'submitted'`** — visible feedback in the gap between submit and first chunk. ~2h.
5. **Slash-command palette** — `/clear`, `/regenerate`, `/canonical`, `/help`. ~1 day.
6. **Model picker in chat header** — replace static `headerMeta` with a `<select>` bound to BE `/api/admin/ai/models`. ~1 day.
7. **Generic `data-*` registry** — factor `readDataPartField` into a typed dispatcher so adding a 4th custom data part doesn't require touching the adapter file. ~4h.
8. **`onData` consumption for transient events** — emit BE-side transient parts (`progress`, `retrieval-debug`) and surface in the dev panel only when a debug flag is on. ~1 day.

### Tier 3 — Specialised / niche

1. **`useObject`** — for the promotion-candidate validation drafting flow; partial JSON streams of the canonical schema. Niche, high-effort.
2. **Tool-calling end-to-end** — `tool-<name>`, `addToolResult`, `addToolApprovalResponse`, `sendAutomaticallyWhen`. AskMyDocs currently has zero tool-calling on the chat path; would only matter if Lorenzo decides chat should expose KB-admin actions inline.
3. **Artifact panel** — large product surface (`vercel/ai-chatbot` ships ~15 components for it). Only if AskMyDocs wants ChatGPT-Canvas-style co-editing.
4. **Public conversation visibility toggle** — share read-only chat URL. Out-of-domain for an internal KB tool.
5. **`TextStreamChatTransport`** — only if a non-UIMessage plain-text endpoint is ever exposed.
6. **`useCompletion`** — inline rewrite UIs; nothing in AskMyDocs needs it today.

## Section 4 — Concrete sample recommendations (3 Tier-1 picks)

### 4.1 Wire `regenerate` (one-line fix, full Tier-1 ROI)

**What the user sees:** every assistant bubble grows a small replay icon in its action row. Click → the bubble dissolves, the spinner re-engages, and a fresh answer streams in over the same retrieval payload. The most common ChatGPT shortcut, and the one feature users notice missing within 30 seconds of using AskMyDocs.

**Code change:** in `frontend/src/features/chat/MessageBubble.tsx` add an `onRegenerate?: () => void` prop, pass it through to `<MessageActions onRegenerate={onRegenerate} />`, and thread `chat.regenerate` from `ChatView.tsx:112` → `MessageThread` → `MessageBubble`. The button DOM (`chat-message-regenerate` testid + ARIA) already exists at `MessageActions.tsx:40-50`. Adds one Playwright scenario verifying the button is reachable + that clicking it issues a fresh stream POST.

### 4.2 `experimental_throttle: 50`

**What the user sees:** on long answers (300+ tokens), the streaming text scrolls smoothly instead of micro-jittering as React reconciles every token. Subtle but noticeable on slower machines and on iOS Safari. Pairs nicely with the existing auto-scroll behaviour in `MessageThread.tsx:83-97` (eliminates the "smooth scroll cancelled mid-flight" complaint already commented on at `MessageThread.tsx:84-95`).

**Code change:** one new prop in `useChat()` call at `use-chat-stream.ts:262-280`. Add an INP measurement in the existing Playwright streaming spec to lock the improvement in CI. Zero risk to the BE wire format.

### 4.3 Suggested follow-up actions

**What the user sees:** after every assistant turn finishes, the empty area under the message displays 2-3 contextually-relevant follow-up prompts ("Show me the policy for new hires", "Compare with the remote-work doc", "Why is this canonical?"). Clicking dispatches `sendMessage` with that text. Today the only suggestion surface is the three static buttons on the `EmptyThread` card, which disappear after the first turn.

**Code change:** BE adds a `StreamChunk::dataSuggestions(array $prompts)` emitter that fires once at stream end (cheap — re-uses the LLM context). FE adapter reads `part.type === 'data-suggestions'`, renders an `<SuggestedActions>` row below the last assistant bubble in `MessageThread.tsx`, click dispatches `chat.sendMessage({ text: prompt })`. Mirrors `vercel/ai-chatbot/components/chat/suggested-actions.tsx`. End-to-end Playwright covers happy path (prompt rendered + click triggers new turn) + empty path (BE returns zero suggestions → row absent).
