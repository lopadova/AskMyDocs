## Architectural outcome

```
v5.0 (retired):
   Host PHP → POST localhost:3535/invoke-tool → Node sidecar → upstream MCP server
                                                  (TypeScript)

v7.0:
   Host PHP → McpClient::forServer($adapter) → upstream MCP server
              (padosoft/askmydocs-mcp-pack)     (HTTP / SSE / stdio)
```

The Node sidecar — its Dockerfile, supervisord config, TypeScript clients,
fixtures, and jest tests — is fully removed. Operators no longer manage a
separate process; deployment scripts no longer wait for the sidecar's
`/healthz` to come up; the cURL→Node→cURL hop is eliminated.

## Contract preservation

The W6 integration is invisible to downstream callers. Specifically:

- **`mcp_tool_call_audit`**: schema is now a superset of v5's. The package
  writer fills `input_hash` (canonical SHA-256 over `(tool_name, arguments)`)
  + `actor` (opaque string) + `mcp_server_name` (denormalised) on every
  row; the host's existing writer keeps populating `user_id`,
  `mcp_server_id`, `conversation_id`, `message_id` verbatim. The `status`
  column widened from a strict ENUM to `varchar(32)` so `transport_error`
  (the package's new non-timeout transport-failure marker) accepts cleanly
  — the host audit constants now include `STATUS_TRANSPORT_ERROR`.

- **`mcp_servers.handshake_response_json`**: still emits the canonical
  `{ok, status, protocol_version, server_info: {name, version}, capabilities,
  tools, duration_ms}` shape the admin FE (`HandshakeStatus.tsx`,
  `ToolMatrix.tsx`) has read since v5.0. v7.0/W6.3.B flattened the MCP
  spec's camelCase `initialize` response into those snake_case top-level
  keys + adds a `tools_list_warning` field (additive per R27) when
  `tools/list` failed but `initialize` succeeded — preserving the legacy
  sidecar's non-fatal-tools semantics.

- **Admin REST endpoints**: `GET /api/admin/mcp-tool-call-audit` validator
  + response shape were aligned with the SPA in W6.3.A. Filter param
  `server_id` (SPA-canonical) now works alongside `mcp_server_id` (legacy);
  pagination shipped via `page` / `per_page` + `meta.{total, per_page,
  current_page, last_page}`. Row payload carries flat `user_id`,
  `user_name`, `mcp_server_id`, `mcp_server_name` for the SPA columns
  AND the nested `user` / `mcp_server` keys for any non-FE consumer.

- **DSAR (GDPR Art. 15 / 17)**: `AskMyDocsUserDataExporter` +
  `AskMyDocsUserDataDeleter` now match `mcp_tool_call_audit` rows by the
  legacy `user_id` join AND by every common `actor` shape the package
  may write (`(string)$id`, `user:$id`, email, `user:$email`). Audit
  rows written by the package post-cutover are correctly included in
  exports / erased on deletion.

## Security improvements

- **`/api/mcp/credentials` removed** (W6.3.B). The endpoint was the Node
  sidecar's bootstrap hook for decrypted upstream `auth_config`. With
  the sidecar retired and `MCP_INTERNAL_AUTH_TOKEN` defaulting to empty,
  it was reachable by any authenticated user — a latent decrypted-secret
  pathway. The route is gone, the controller method is gone.
- **`/api/mcp/internal-auth` + `MCP_INTERNAL_AUTH_TOKEN` removed**
  (W6.3.C). The surviving sidecar token-presence probe + its companion
  env var + `mcp.internal_auth_token` config + the entire
  `McpInternalAuthController` were all retired. After PR #179 lands
  there is no host-side residue from the v5.0 Node sidecar era at all.
  Both retired routes are smoke-tested as 404 in
  `frontend/e2e/admin-mcp-super-admin.spec.ts` so a silent restoration
  fails CI loudly.
- **Header case-insensitivity + Bearer-token hygiene** in
  `McpServerAdapter::extractHeaders()` — operator-stored `headers.authorization`
  (lowercase) no longer duplicates with a synthesised `Authorization`;
  empty / whitespace-only `auth.token` no longer ships `Bearer ` (no
  credential, masks the real config error).
- **Unconfigured-tools guard** in `EloquentMcpServerRegistry::hasConfiguredTools()`:
  rows with `null` / `[]` / garbage `enabled_tools_json` no longer reach
  the package as "wildcard, all tools enabled" (which the package
  contract would treat as full access). Only explicit `['*']` or a
  list with at least one trimmed-non-empty string entry surfaces.

## What W6.3.C closes (PR #179, post-rc1, pre-GA)

The W6 charter originally listed "switch 3 controller imports + delete
6 inline files". As W6.3.B / W6.3.C unfolded that scope was split into
two streams with different urgency:

1. **Sidecar-artefact retirement** — load-bearing for v7.0 because it
   eliminates dead host-side attack surface (decrypted-secret callbacks,
   tokens that gate nothing). W6.3.B retired the Node sidecar itself +
   `McpClientBridge` + `/api/mcp/credentials`; **W6.3.C (PR #179)**
   retires the surviving `/api/mcp/internal-auth` probe + the
   `MCP_INTERNAL_AUTH_TOKEN` env var + the `mcp.internal_auth_token`
   config key + `McpInternalAuthController`. After PR #179 lands the
   sidecar-retirement story has zero remaining surface area: no
   internal callbacks, no decrypted-secret pathways, no token guard
   on a dead route.

2. **Inline orchestrator consolidation** — NOT load-bearing for v7.0
   because the inline `McpToolCallingService` / `McpServerRegistry` /
   `McpToolAuthorizer` already run on native MCP transports (W6.3.B
   rewrote `ToolInvoker` + `McpHandshakeService` to drive
   `McpClient::forServer()` underneath). The host-specific telemetry
   the inline stack carries (per-turn tool-call summary with
   status/server/result metadata) doesn't have a direct equivalent on
   the package's `HostChatResponse`; consolidation needs a thin
   translation adapter that the GA timeline didn't justify cutting
   short. **Deferred to a post-v7.0 cycle.**

The remaining inline `app/Mcp/Client/*` files are therefore intentional
augmentation, not technical debt — they run on the same native
transports as the package's own services, just with richer host
telemetry on top.
